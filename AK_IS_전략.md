> "계열사 비즈니스가 멈추지 않게 시스템을 안정적으로 운영하면서, 점진적으로 고도화·전환할 수 있는 IT 인력"

| 이것이 아님 | 이것이 맞음 |
|------------|-----------|
| 화려한 서비스 개발자 ❌ | 운영 기반 + 개선·확장 가능한 개발자 ⭕ |
| 연구 중심 R&D ❌ | |

## 2. AK IS 신입에게 기대하는 3가지 핵심 역할

### ① SM 관점 (운영 안정성)

- ERP, 내부 업무 시스템 장애 없이 돌리는 능력
- 로그 분석, 원인 파악, 재발 방지
- **"시스템을 이해하고 책임지는 사람"**
- 신입에게도 '안정성 마인드' 매우 중요

### ② SI 관점 (확장·연계)

- ERP(SAP) + 외부 시스템 연동
- REST API, DB, 배치, 인터페이스 이해
- **"시스템 간 연결 고리" 역할**

### ③ 디지털 전환 관점 (미래 포석)

- 클라우드 전환
- AI / 자동화 PoC
- **"지금은 작게, 나중엔 크게 쓸 수 있는 기술 감각"**

---

## 3. 이력 → AK IS 직무 연결 포지셔닝

### 캐릭터 한 줄 정의

> "운영 안정성을 이해하는 시스템형 개발자 + 자동화를 고민하는 실무형 인재"

**핵심**: AK IS는 '천재 개발자'보다 **'믿고 맡길 수 있는 사람'**을 원함

---

### 이력 매칭 구조

#### ① SCS Pro (임베디드 / 결제 단말기) → 운영 관점 최강 카드

- 단말기 = 장애 나면 바로 매출 영향
- 하드웨어–펌웨어–서버 연동 경험
- **안정성 / 테스트 / 책임감 강조**

**이렇게 풀어라:**
> "결제 단말기 특성상 장애가 곧 비즈니스 중단으로 이어지기 때문에,
> 운영 안정성과 재현 가능한 문제 해결을 가장 중요하게 다뤄왔습니다."

➡ SM/운영 직무에 딱 맞는 언어

---

#### ② Full-stack 프로젝트 경험 → SI 인력으로서의 확장성 증명

- DB 설계
- REST API
- 프론트–백엔드 연계
- 요구사항 → 구현 → 테스트

**이렇게 말해라:**
> "기능 구현뿐 아니라, 유지보수와 확장까지 고려한 구조를 고민했습니다."

➡ SI + ERP 연계 쪽으로 자연스럽게 연결

---

#### ③ AI / Agentic AI 활용 경험 → 디지털 혁신 '씨앗' 역할

**절대 하면 안 되는 말:**
- ❌ "LLM 모델 연구했습니다"
- ❌ "최신 AI 기술에 관심 많습니다"

**이렇게 말해야 함:**
> "운영 업무에서 반복되는 작업을 줄이기 위해
> AI를 '보조 도구'로 적용하는 방식에 관심을 가져왔습니다."

**적용 예시:**
- 로그 요약
- 장애 리포트 자동 생성
- 고객 문의 분류
- 정산/문서 자동화

➡ AK IS가 딱 원하는 '현실적인 AI 활용'

## 7. 문항별 상세 전략 설계

### [문항 1] 자유 소개 – 구조 설계도

**HR은 이렇게 읽는다:**
- 이 사람, 조직에서 튀지 않나?
- 말이 현실적인가?
- 책임 회피형인가, 책임 수용형인가?
- 운영 조직에 버텨낼 성향인가?

❗ **여기서는 기술 깊이 전혀 안 본다.**

**전략 핵심:**
- 기술 = 배경
- 태도 = 주인공

**권장 구조 (AK IS 합격자 구조):**

| 순서 | 내용 |
|------|------|
| 1단락 | 개발관 선언 - "나는 어떤 기준으로 개발하는 사람인가" |
| 2~3단락 | 운영 경험 스토리 - 장애 / 안정성 / 책임 / 검증 |
| 4단락 | AK IS와의 접점 - 그룹 IT / ERP / 운영 중심 |
| 마무리 | 입사 후 태도 - 빠른 이해 → 안정적 기여 |

**필수 사고 프레임:**
1. "문제가 생겼을 때"
2. "원인을 파악했고"
3. "다시 발생하지 않도록"

> 이 3단계 사고는 SM/SI 조직에서 신입 최고 가치

**주의:** AI, 클라우드는 '주연' 금지
- "도입했다" ❌
- "활용 가능성을 고민했다" ⭕

---

### [문항 2] 직무 관련 교육 및 경험 – 구조 설계도

**HR + 실무 리더가 이렇게 읽는다:**
- 업무 구조 이해 가능?
- ERP/DB/API 개념은 아나?
- 운영하면서 버틸 수 있나?
- 교육하면 커질 여지가 있나?

**권장 구조:**

| 순서 | 내용 |
|------|------|
| 1단계 | 전공 / 교육 요약 - DB, OS, 네트워크, SW공학 |
| 2단계 | 시스템 구축 경험 - 웹/서버/DB/연동 |
| 3단계 | 운영 중심 경험 (핵심) - 장애 대응, 로그 분석, 테스트 |
| 4단계 | AK IS 직무로의 연결 - ERP / API / 클라우드 전환 |

**중요 포인트:**
- ❌ "무엇을 배웠다"
- ⭕ "어떻게 쓰일 수 있다"

**연결 예시:**
| 내 경험 | AK IS 연결 |
|--------|-----------|
| REST API | 계열사 시스템 연동 |
| DB | ERP 데이터 정합성 |
| 클라우드 | 운영 효율 개선 |

### 핵심 평가 포인트

- ✔ "이 친구, 현업에 던져도 사고 안 치겠다"는 신뢰감

---

저는 시스템을 '만드는 것'보다, 멈추지 않게 운영하고 책임지는 과정 속에서 개발자의 가치를 실현해 온 인재입니다. 대학 시절 컴퓨터공학을 전공하며 다양한 프로젝트를 수행했지만, 단순한 기능 구현보다 시스템 전체 흐름을 이해하고 안정적으로 유지하는 역할에 자연스럽게 관심을 갖게 되었습니다.

대학교 재학 중에는 AI·데이터 기반 동아리 활동과 팀 프로젝트를 통해 웹 및 서버 기반 시스템을 설계하고 구현하는 경험을 쌓았습니다. 이 과정에서 프론트엔드와 백엔드를 연결하는 API 구조, 데이터베이스 설계, 사용자 요구사항 변경에 따른 유지보수의 중요성을 체감했습니다. 특히 프로젝트가 고도화될수록 '잘 만든 기능'보다 '오래 유지될 수 있는 구조'가 중요하다는 점을 배웠습니다.

이후 현재 재직 중인 SCS Pro에서는 결제 단말기 펌웨어 및 임베디드 시스템을 다루는 업무를 수행하고 있습니다. 결제 시스템은 단 한 번의 오류도 곧바로 매출과 고객 신뢰에 영향을 미치기 때문에, 저는 개발 과정에서 기능 구현보다 안정성 검증, 장애 재현, 재발 방지에 더 많은 시간을 투자해 왔습니다. 로그 분석을 통한 원인 파악, 테스트 시나리오 보완, 현장 환경을 고려한 예외 처리 등은 제 업무의 일상적인 부분이었습니다. 이 경험을 통해 저는 '운영을 고려하지 않은 개발은 완성된 개발이 아니다'라는 확고한 기준을 갖게 되었습니다.

AK IS는 애경그룹의 다양한 계열사 시스템을 안정적으로 운영함과 동시에, ERP 고도화와 클라우드 전환, 디지털 혁신을 추진하는 조직입니다. 저는 이러한 환경이 단순히 새로운 기술을 도입하는 것이 아니라, 기존 시스템을 정확히 이해하고 점진적으로 개선해 나가는 인재를 필요로 한다고 생각합니다. 이는 제가 지금까지 경험해 온 시스템 운영 중심의 개발 방향성과 정확히 맞닿아 있습니다.

입사 후에는 먼저 운영 업무를 통해 시스템 구조와 업무 흐름을 충분히 이해하고, 장기적으로는 반복 업무의 자동화, 시스템 간 연계 개선 등 작지만 실질적인 개선을 지속적으로 만들어내는 IT 인력으로 성장하고 싶습니다. AK IS가 추구하는 'Digital Transformation Partner'로서의 역할을 함께하겠습니다.
