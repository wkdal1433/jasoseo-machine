## Episode 01. 3D Gaussian Splatting 관련 원본 기록

### 프로젝트 시작 배경
- 대학교 '인공지능2' 과목 (2024.09~2025.02, 기말 프로젝트, 2인 팀, A+)
- 대다수가 생성형 AI의 단순 API 활용에 그칠 때 최신 기술인 3D Gaussian Splatting에 주목
- 남들이 GPT API만 쓸 때, 오픈소스 3D AI 기술을 분석해 자체 서버(On-Premise)에 구축하기로 결정
- 단순히 이론에 그치지 않고 학우들이 이 기술을 직접 체감할 수 있도록 서비스화하기로 결정

### 의사결정 지점들 (대안 비교)
- **경량화 모델 선택**: 다양한 경량화 모델(Mini Splatting, Compact3D 등)을 비교·테스트 → Mini Splatting 선택 (이유: 연산 시간 대비 품질 최적)
- **배포 방식 선택**: Cloudflare CDN vs 자체 서버 → Cloudflare 시도 후 실패 → 같은 네트워크 내 자체 서버로 전환 (이유: 대용량 PLY 파일의 캐싱 정책 충돌)
- **해상도 제한 결정**: 입력 영상 해상도를 제한하여 추가 최적화 수행 (품질과 속도의 트레이드오프)

### 시행착오 기록
- GPU 메모리 부족 문제 → Mini Splatting 도입 결정
- 초기에는 학교 서버 GPU 한계로 모델링 시간이 과도하게 오래 걸렸다
- 모델링 시간을 20분에서 1~2분으로 단축
- 입력 영상 해상도 제한을 통해 추가 최적화를 수행
- Colmap 전처리 단계의 병목은 당시 지식의 한계로 해결하지 못함
- Cloudflare를 활용해 결과물을 제공하려 했으나 실패:
  - 대용량 산출물의 캐싱 정책 충돌 발생
  - 수동 빌드/업로드 이슈 발생
  - 모니터링 체계 부재
- 결국 같은 네트워크 내 학교 컴퓨터를 직접 서버로 띄워 파이프라인을 운영하는 방식으로 전환

### 기술적 구현 상세
- 학교 실습실 컴퓨터에 직접 오픈소스 모델을 구축하여 렌더링 서버를 띄움
- 프론트엔드 뷰어까지 개발
- COLMAP 전처리 → Gaussian Splatting → PLY 웹뷰어 파이프라인 구축
- Mini Splatting 활용 경량화
- Java 백엔드 구현
- 영상 기반 3D 모델링 웹 파이프라인 개발
- 밤낮으로 논문을 분석하고 여러 경량화 모델을 비교·테스트

### 결과 및 성과
- 학우 대상 3D 체험 서비스 런칭
- A+ 성적 획득 (인공지능2 과목)
- 웹 뷰어 시연 성공
- 사진만으로 3D 환경을 구현하는 기술 완성

### 후속 활동
- AI & Applications Lab (명지대학교 랩실, 2025.01 ~ 2026.01)
- 컴퓨터비전 연구실에서 6개월간 시간축 추가 확장 연구를 진행
- 2인 팀으로 가설 설정부터 검증까지 전 과정을 수행하며 업무가 과중됨
- AI 연구의 깊이와 실무 적용의 속도감·효율성 사이에서 고민
- AI 기술의 진정한 가치는 연구 그 자체가 아니라 실제 문제 해결과 생산성 혁신에 있음을 깨달음

### 깨달음
- AI 파이프라인은 연산, 저장, 배포, 모니터링이 유기적으로 연결된 구조라는 점을 깨달음
- AI 파이프라인의 운영은 클라우드 관점의 인프라 이해가 필수임을 절실히 체감
- AI/Data 파이프라인을 운영하려면 계산, 저장, 배포, 관리가 모두 연결되어야 함을 깨달음

### [AI 참조용] 자소서 활용 메타데이터
- **앵글 시프팅 가능 방향**:
  - Execution: "연구용 오픈소스를 사용자 서비스 구조로 전환한 실행력"
  - Growth: "최신 AI 논문을 분석하고 실제 서비스로 구현한 학습력"
  - Stability: "배포 실패 후 대안을 찾아 안정적 파이프라인 구축"
  - Communication: 해당 없음 (1~2인 프로젝트)
- **이중 코딩 연결 가능 가치**: 도전, 혁신, 안정성, 기술 깊이, 문제해결
- **구체성 기울기 재료**: 모델링 시간 20분→1~2분, 경량화 모델 비교 테스트, Cloudflare 실패→자체 서버 전환, A+ 성적
- **답변 가능 질문 유형**: 도전/몰입 경험, 기술적 문제해결, 어려움 극복, 성장 경험, 지원 동기(AI/DX 직무)

## Episode 02. DayScript 관련 원본 기록

### 프로젝트 시작 배경
- AI 코딩 시대에 대한 개인적 관찰
- AI Agent(Claude Code)와 바이브 코딩(Vibe Coding)이라는 개발 패러다임의 변화를 빠르게 포착
- "개발 지식을 갖춘 기획자가 AI와 협업할 때 생산성의 한계는 없다"는 가설을 검증하기 위해 DayScript 프로젝트를 주도
- Figma 기획서 별도 제작 (2025.09~2025.11, 1인, Figma 작업물 존재)

### 개발 과정
- 기획 직후 AI와 협업하여 TypeScript 기반 로직을 생성
- 실시간 피드백으로 설계를 수정하는 초단기 애자일 프로세스를 확립
- Figma 설계부터 구현까지의 전 과정을 압도적인 속도로 완주
- Claude Code로 목업 디자인을 생성
- Figma에서 조합해 색상과 배치를 조정하며 디자인을 완성
- 기존에 몇 주가 걸리던 작업을 일주일 내로 단축
- Figma MCP로 디자인 링크를 연동해 Claude Code로 구현
- GitHub MCP로 기능별 커밋을 자동 관리
- PM처럼 오류 처리와 수정 사항만 결정하는 워크플로우를 구축
- React Native 기반의 안드로이드 구현은 처음임에도 기존 안드로이드 스튜디오 작업 방식보다 훨씬 효율적으로 진행
- Figma/GitHub/Notion MCP 연동
- 바이브 코딩 워크플로우 구축
- 레벨/포인트 시스템, 학습 기록 관리 구현

### 의사결정 지점들
- **AI 도구 선택**: Claude Code를 선택한 이유 (MCP 연동으로 기획→디자인→구현 파이프라인 자동화 가능)
- **프레임워크 선택**: React Native 선택 (크로스 플랫폼 + AI 코딩 친화적)
- **워크플로우 설계**: 직접 코딩 vs PM형 감독 → PM형 감독 선택 (AI에 구현 위임, 오류/방향만 관리)

### 시행착오 기록
- Claude Code 초기 사용 시 어려웠던 점
- MCP 설정 과정의 삽질
- 디자인과 백엔드/프론트 코드까지는 검증하며 구축할 수 있었지만 클라우드 MCP 워크플로우는 클라우드 관련 지식이 부족해 완성하지 못한 점이 아쉬움

### 성과
- 기획-설계-구현 사이클 대폭 단축
- 1인 개발 생산성 증명
- 기존 수 주가 걸리던 개발 과정을 1주일로 단축하며 실무 효율화를 증명
- AI 에이전트를 활용한 바이브 코딩 워크플로우로 개발 효율을 70% 이상 향상

### [AI 참조용] 자소서 활용 메타데이터
- **앵글 시프팅 가능 방향**:
  - Execution: "가설→검증→실증까지 완주한 실행력"
  - Growth: "AI 패러다임 변화를 빠르게 포착하고 학습한 흡수력"
  - Stability: 해당 없음 (실험적 프로젝트)
  - Communication: "기획자 관점에서 AI와 협업하는 새로운 소통 모델"
- **이중 코딩 연결 가능 가치**: 생산성 혁신, 자기주도, 도전, 기술 트렌드 선도
- **구체성 기울기 재료**: 개발 기간 수 주→1주일, 효율 70% 이상 향상, Figma/GitHub/Notion MCP 3종 연동
- **답변 가능 질문 유형**: 도전/혁신 경험, 자기주도 프로젝트, AI 활용 역량, 생산성 향상 사례, 기획/개발 융합

## Episode 03. 코드비전 관련 원본 기록

### 근무 정보
- (주)코드비전 데이터 라벨링 실무
- 2024.10.12 ~ 2025.06.07 (약 9개월)
- AI 학습용 데이터셋 구축 및 품질 검수 (QA)
- 계약직 / 데이터 라벨러

### GIGO 원칙을 체감한 순간
- AI 모델 성능이 데이터 품질(GIGO)에 크게 의존함을 실무에서 체감
- 데이터 라벨러로 근무하며 AI 학습용 데이터의 정합성을 검증할 때, 작은 오류가 모델 성능을 저하시키는 것을 봄
- 고객 데이터 하나하나가 서비스 품질을 결정한다는 '무결성의 법칙'을 체득

### 검수 업무 상세
- 1px 단위 오차, 태깅 불일치, 라벨 노이즈까지 집요하게 검수
- 학습 데이터의 오류·노이즈를 최소화하여 모델 신뢰도 확보
- 1원의 오차, 1px의 오류도 허용하지 않는 데이터 검수 집요함 (QA 마인드)

### 깨달음
- "Garbage In, Garbage Out"이라는 데이터의 본질을 깨달음
- 이 가치를 실무 품질 검수(QA) 역량으로 발전시킴
- 데이터 무결성(Data Integrity)은 모든 IT 시스템의 근본임을 체감

### [AI 참조용] 자소서 활용 메타데이터
- **앵글 시프팅 가능 방향**:
  - Execution: "실무 환경에서 데이터 품질을 보장한 검수 역량"
  - Growth: 해당 없음 (실무 적용 중심)
  - Stability: "1px 단위 검수 → 리스크 관리형 인재"
  - Communication: 해당 없음 (개인 업무 중심)
- **이중 코딩 연결 가능 가치**: 신뢰, 품질, 꼼꼼함, 책임감, 고객 안심, 데이터 무결성
- **구체성 기울기 재료**: 9개월 근무, 1px 단위 검수, AI 모델 성능-데이터 품질 상관관계
- **답변 가능 질문 유형**: 직무 경험, 책임감/꼼꼼함, 가치관, 데이터/품질 관련 역량

## Episode 04. MediaPolytech 관련 원본 기록

### 프로젝트 정보
- (주)MediaPolytech 외주 백엔드 프로젝트
- 2024.10 ~ 2025.08
- 백엔드 리드 (기여도 40%)
- 서비스 기획/디자인(Figma)부터 프론트엔드, 백엔드, 서버 배포 단계까지 전 과정에 주도적으로 참여하여 프로젝트 완수
- **캐나다 체류 중 진행** → 시차를 Git 기반 비동기 협업으로 극복하며 납기 내 런칭

### 로컬 → S3 전환을 결정한 계기
- 로컬 저장소의 한계 및 보안 취약점 발견
- 안정적인 인프라가 서비스의 확장성을 결정짓는 핵심 요소임을 뼈저리게 경험
- 이미지와 영상 저장 방식을 분리해 S3 비용을 절감

### 의사결정 지점들
- **저장소 전환**: 로컬 저장소 vs AWS S3 → S3 선택 (이유: 보안 취약점 + 확장성 한계)
- **이미지/영상 분리**: S3 내에서 이미지와 영상 저장 체계를 분리 (이유: 비용 절감 + 데이터 관리 효율)
- **설계 원칙**: 빠른 구현 vs SOLID 원칙 적용 → SOLID 선택 (이유: 장기 유지보수성 우선)
- **보안 구현**: 외부 라이브러리 vs 직접 구현 → Spring Security + JWT 직접 구현 (이유: 보안 흐름 완전한 이해)

### SOLID 원칙 적용
- 서비스 확장성에 대한 부담을 줄이기 위해 SOLID 원칙을 학습해 설계에 적용
- 관리자 CRUD 구현 시 파일 업로드·삭제 전용 FileService로 책임을 분리 (SRP)
- 게시물별로 달라지는 파일 처리 로직은 전략 패턴으로 위임해 구조적 복잡성을 해결 (OCP)
- Spring Boot 환경에서 SOLID 원칙을 적용해 코드를 리팩토링

### 보안 구현
- Spring Security + JWT 기반 인증/인가 시스템 구축
- 관리자 인증·인가를 처리하기 위해 Spring Security와 JWT 기반 보안 필터를 직접 구현
- 직접 구현한 필터가 관리자 요청의 인증·인가를 처리하도록 구성
- AccessToken/RefreshToken 분리 및 RBAC로 API 엔드포인트별 보안 강화

### 구현 기능
- 공지사항·문의사항·제품·설치사례 등 주요 기능 담당
- 회원, 공지사항, 문의사항, 제품, 설치사례 테이블 설계 및 CRUD
- 페이징, 키워드 검색 구현

### 팀 협업 이슈
- 프론트엔드 A, Quill 에디터 연동 B와 3인 팀을 구성
- B 파트 지연으로 두 달간 일정이 막히며 분위기가 경직됨
- 함께 검토한 결과 S3 API 함수명 불일치가 문제였음을 확인하고 즉시 수정
- 이후에도 진행이 더뎌, A와 함께 나머지 관리자 페이지를 직접 구현

## 자기 인식 / 가치관 관련 기록

### 직장 선택 기준
- 첫째, 성장 가능성이 큰 산업에서 미래를 주도하는 기업인지
- 둘째, 보유한 기술이 비즈니스와 사회에 실질적 가치를 만들어낼 수 있는 환경인지

### 개발자 역량에 대한 생각
- 앞으로 개발자 역량은 세 가지로 정의된다고 생각
- 첫째 최신 AI 기술을 이해하고 활용하는 능력
- 둘째 AI와 협업해 백엔드/프론트엔드를 구현하는 능력
- 셋째 클라우드를 기반으로 안정적이고 확장 가능한 시스템을 운영하는 능력
- 클라우드 전문성은 실제 서비스 운영 경험 없이는 습득하기 어려운 요소로, 미래 개발자를 구분할 핵심 요소

### 한계 인식
- 더 이상 백엔드/프론트엔드만 다루는 개발자로는 AI 기술의 속도를 따라갈 수 없다
- 배포 단계에서는 AWS를 깊이 이해하지 못한 채 다른 사람들을 따라하는 수준에 머물렀다
- 생성형 AI, AI 에이전트, MCP 순으로 개발의 경계가 무너지는 변화를 겪으며 '단순 웹 개발자'로는 한계가 명확하다는 사실을 인식
- MSA 설계, CI/CD 자동화, 클라우드 모니터링 등까지는 직접 처리할 수 없다는 한계를 느낌
