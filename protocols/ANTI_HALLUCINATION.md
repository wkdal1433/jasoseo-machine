# Anti-Hallucination Protocol
# 할루시네이션 방지 상세 규칙

> **이 규칙은 모든 자기소개서 응답에 절대적으로 적용된다.**
> 위반 시 허위 사실 기재로 간주되며, **면접 탈락·신뢰 상실·법적 리스크**를 초래한다.

---

## 1. 핵심 원칙: Episode = 유일한 사실 원천

| 규칙 | 설명 |
|------|------|
| **Single Source of Truth** | 자기소개서에 사용할 수 있는 정보 = Episode 파일에 명시된 내용 ONLY |
| **Zero-Assumption** | Episode에 명시되지 않은 정보 = 존재하지 않는 사실 |
| **Quote-Level Fidelity** | 고유명사·수치·기간·기술명은 문자 그대로 일치 |

---

## 2. Zero-Assumption Rule (무가정 원칙)

**Episode 파일에 "명시적으로 서술되지 않은 모든 정보"는 설령 논리적으로 타당하더라도 사용 불가**

- 암묵적 역할 ❌
- 업계 관행 ❌
- 전후 맥락 보완 ❌
- 기억 기반 재구성 ❌
- 맥락상 "당연히 했을 것"이라는 추론 ❌
- 비슷한 경험으로 대체 ❌

---

## 3. 절대 금지 행위

| 유형 | 설명 | 예시 |
|------|------|------|
| **명칭 변조** | 역할·직책·프로젝트명 임의 변경 | 스터디장 → 파트장 |
| **기간/기수 변조** | 활동 기간·기수 변경 | 5기 → 6기 |
| **기술 스택 추가** | Episode에 없는 기술 삽입 | 없는 React / Node.js |
| **수치 변조** | 성과 수치 수정·반올림 | 85% → 90% |
| **경험 창작** | Episode에 없는 경험 생성 | 존재하지 않는 프로젝트 |
| **맥락 왜곡** | S/T/A/R 의미 변경 | 문제·행동·결과 재해석 |
| **표현 강화** | 사실 과장·미화 | "주도적으로", "전사적" |

---

## 4. 작성 전 필수 검증 프로토콜

1. 해당 Episode 파일을 **직접 열어 확인** (사전 기억·이전 대화 의존 금지)
2. 문장 단위로 **사용 가능한 정보만 추출**
3. 추출된 정보만으로 **문장 구성**
4. Episode에 없는 정보는 **단 한 글자도 추가 금지**

---

## 5. 문장별 자가 검증 체크리스트

| 검증 항목 | 자가 질문 |
|----------|----------|
| **존재 여부** | 이 정보가 Episode에 문자 그대로 존재하는가? |
| **명칭 일치** | 역할·프로젝트명·기술명이 완전 동일한가? |
| **수치 일치** | 모든 숫자·기간·비율이 원본과 일치하는가? |
| **맥락 보존** | Episode의 의미를 확장·축소·재해석하지 않았는가? |

> **→ 하나라도 "100% Yes"가 아니면 작성 중단**

---

## 6. 불확실성 발생 시 (Fail-Safe)

**DO:**
- "Episode 파일에서 [X]에 대한 명시적 정보가 없습니다."
- "이 내용을 추가하려면 Episode 보완이 필요합니다."
- 사용자 확인 후에만 진행

**DON'T:**
- 추론으로 보완
- 유사 경험으로 대체
- 일반적인 표현으로 흐리기

---

## 7. 최종 제출 전 Hard Gate

**아래 전 항목 통과 필수:**

- [ ] 모든 프로젝트명이 Episode와 철자까지 동일한가?
- [ ] 모든 역할·직책 명칭이 Episode와 정확히 일치하는가?
- [ ] 모든 기간(년/월)·기수가 Episode와 완전히 동일한가?
- [ ] 모든 기술 스택이 Episode에 명시적으로 존재하는가?
- [ ] 모든 정량 성과가 원본 수치 그대로인가?
- [ ] Episode에 없는 경험·표현이 단 한 문장도 없는가?
- [ ] STAR 구조가 원본 의미를 왜곡하지 않았는가?

> **→ 하나라도 불합격: 해당 문장 삭제 또는 Episode 재확인 후 재작성**
