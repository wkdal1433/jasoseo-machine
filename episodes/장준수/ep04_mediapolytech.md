## S-P-A-A-R-L 구조

### S (상황)
- (주)MediaPolytech 외주 백엔드 프로젝트
- 서비스 기획/디자인(Figma)부터 프론트엔드, 백엔드, 서버 배포까지 전 과정에 주도적 참여
- 프론트엔드 A, Quill 에디터 연동 B와 3인 팀 구성

### P (문제 정의)
- 로컬 저장소의 한계 및 보안 취약점 발견
- 팀 프로젝트가 커지면서 코드 유지보수 어려움 증가
- B 파트 지연으로 두 달간 일정이 막히며 분위기 경직

### A (분석 / 의사결정)
1. **저장소 전환**: 로컬 vs AWS S3 → S3 선택 (이유: 보안 취약점 + 확장성 한계)
2. **이미지/영상 분리**: S3 내에서 저장 체계를 분리 (이유: 비용 절감 + 데이터 관리 효율)
3. **설계 원칙**: 빠른 구현 vs SOLID 원칙 → SOLID 선택 (이유: 장기 유지보수성 우선)
4. **보안 방식**: 외부 라이브러리 vs 직접 구현 → Spring Security + JWT 직접 구현 (이유: 보안 흐름의 완전한 이해)
5. **팀 이슈 대응**: B 파트 지연 원인 공동 검토 → S3 API 함수명 불일치 확인·수정 → 이후에도 지연 시 A와 함께 직접 구현

### A (행동)
- **보안**: Spring Security + JWT 기반 인증/인가 시스템 직접 구현, AccessToken/RefreshToken 분리, RBAC로 API별 보안 강화
- **확장성**: 로컬 저장소 → AWS S3 마이그레이션, 이미지·영상 저장 방식 분리로 비용 절감
- **유지보수**: SOLID 원칙 기반 리팩토링 — 파일 업로드·삭제 전용 FileService로 책임 분리(SRP), 게시물별 파일 처리는 전략 패턴으로 위임(OCP)
- **기능 구현**: 공지사항·문의사항·제품·설치사례 CRUD, 페이징, 키워드 검색
- **팀 이슈 해결**: S3 API 함수명 불일치 확인·수정, A와 함께 나머지 관리자 페이지 직접 구현
- **글로벌 협업**: 캐나다 체류 중 시차를 Git 기반 비동기 협업으로 극복하며 납기 내 런칭

### R (결과)
- 외주 프로젝트 성공적 완수 및 런칭
- 보안성·확장성·유지보수성을 갖춘 백엔드 아키텍처 구축
- 캐나다-한국 간 시차 극복 비동기 협업 성공

### L (배움)
- 뒤늦게 B가 다른 활동으로 프로젝트에 집중하지 못했다는 사실을 알게 됨
- 팀원 상황을 더 살피고 함께 해결해야 했음을 깨달음
- "팀 프로젝트에서 내가 더 소통하고 도우면 내가 더 배운다"는 교훈
- 이후 프로젝트에서는 팀원 상황을 먼저 파악하고 협업하는 태도로 전환
- 안정적인 인프라가 서비스의 확장성을 결정짓는 핵심 요소임을 체감

## 자소서 작성 시 주의사항

### DO (이렇게 써라)
- "로컬 → S3 전환의 **의사결정 과정** (보안 취약점 → 확장성 판단)"
- "SOLID 원칙을 **FileService(SRP), 전략 패턴(OCP)**으로 실제 적용"
- "Spring Security + JWT **보안 필터를 직접 구현**"
- "캐나다 체류 중 **시차를 Git 비동기 협업으로 극복**"
- "B 파트 지연 → **원인 공동 검토 → 직접 해결**한 팀 이슈 대응"

### DON'T (이렇게 쓰지 마라)
- 기술 스택만 나열 (Spring, AWS... 등)
- "백엔드 개발했다" (왜 그 선택을 했는지가 중요)
- 팀 갈등을 "탓"으로 서술 (배움으로 서술해야 함)
