## S-P-A-A-R-L 구조

### S (상황)
- 비전공자/전공자 간 실력 격차 존재
- 기획자/디자이너와 개발자 사이의 소통 오류 빈발
- "기술적으로 안 돼요"라는 말로 대화가 끊기는 상황

### P (문제 정의)
- 파트장 승격 후 안드로이드 1·2반과 서버 스터디까지 관리하면서 **모든 팀원을 직접 지원할 수 없는 한계** 경험
- 획일적인 커리큘럼으로는 비전공자~전공자 모두를 만족시키기 어려움
- "기술적으로 불가능한 기획"과 "비즈니스 맥락을 모르는 개발" 사이의 간극

### A (분석)
- 개인 지원 → 조직 관리 중심으로 전환해야 함을 판단
- 팀원 수준을 3단계로 분류, 각 수준에 맞는 학습 방식 필요
- "안 돼요" 대신 "이렇게 하면 가능하다"는 대안 제시가 핵심

### A (행동)
- **기술 통역**: "안 돼요" 대신 **대안 제시** (왜 안 되는지 + 대신 이렇게 하면 가능)
- **수준별 과제 분리**: 팀원 수준 3단계 분류 — 잘하는 팀원에게는 심화 기술 챌린지, 어려워하는 팀원에게는 코드 리뷰와 1:1 멘토링
- **간접 관리 체계**: 각 스터디장과 주간 회의 진행, 학습 진도 모니터링, 공통 문제와 해결책을 공유하여 스터디장이 팀원을 효과적으로 이끌 수 있는 체계 구축

### R (결과)
- **팀 이탈률 0%**
- **참여율 85%** 달성
- **스터디 완주율 85%** 달성
- 데모데이에서 전체 **10개 팀 중 완성도 상위권**으로 MVP 시연 성공
- **MVP 수상** + **우수상 수상** (리더십 인정)
- 실제 앱 런칭 성공
- 팀원들은 "포기하지 않고 끝까지 할 수 있었다"며 높은 만족도

### L (배움)
- 개인 지원에서 조직 관리 중심으로 전환하며 **조직 전체 성과를 달성하는 리더십** 체득
- "기술의 가능성과 한계를 명확히 알고 조율하는 기획자"의 필요성을 절감
- 각자의 속도를 존중하면서도 공동 목표를 달성하는 **맞춤형 리더십** 확립

## 자소서 작성 시 주의사항

### DO (이렇게 써라)
- "비전공자/전공자 격차를 **수준별 3단계 과제로 해소**"
- "'안 돼요' 대신 **대안을 제시하는 기술 통역자** 역할"
- **정량 성과 필수**: 이탈률 0%, 참여율 85%, 완주율 85%, MVP 수상, 우수상
- "개인 지원→조직 관리로 **리더십 진화** 과정"

### DON'T (이렇게 쓰지 마라)
- "파트장 했다" (직책이 아닌 행동이 중요)
- "협업 경험이 있다" (어떻게 협업했는지가 핵심)
