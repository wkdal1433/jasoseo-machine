# Angle Shifting Map
# 경험 앵글 전환 매핑 (S급 합격자소서 역추적)

> **목적**: 동일인이 같은 경험을 4개 기업에 맞춰 어떻게 다르게 프레이밍했는지 역추적
> **활용**: Episode를 기업별로 재구성할 때의 핵심 전략 가이드
> **원칙**: 사실을 바꾸는 게 아니라, **강조점과 해석을 바꾸는 것**

---

## 1. 앵글 전환 5가지 기법

S급 작성자가 같은 경험을 다른 기업에 맞출 때 사용한 기법:

| 기법 | 정의 | 예시 |
|------|------|------|
| **강조/생략 전환** | 같은 사실에서 어떤 요소를 부각하고 어떤 요소를 축소할지 결정 | KB증권: 기술 디테일 부각 / 삼성생명: 팀 협업 부각 |
| **가치 프레이밍** | 같은 행동을 기업의 가치 언어로 재포장 | "모니터링 구축" → KB: "근본 추적" / 현대: "안심 제공" |
| **문제 재정의** | 같은 문제를 기업이 중시하는 관점으로 재정의 | 한국자금중개: "서버 다운 ≠ 문제, 데이터 비영속성 = 문제" |
| **주체 비중 조절** | "나"의 역할 vs "팀"의 역할 비중을 기업 가치에 맞게 조절 | KB: "저는 직접 구축했고" / 삼성: "팀원과 함께 범위를 좁혀 나갔습니다" |
| **깊이 조절** | 기술적 디테일의 깊이를 기업 성격에 맞게 조절 | KB: HNSW 인덱스까지 명시 / 현대: "벡터 DB의 인덱스"로 축약 |

## 2. 경험별 앵글 전환 상세 매핑

---

### 경험 A: RAG 기반 사내 챗봇 PoC (인턴십)

> **사실**: 인턴십에서 RAG 챗봇 구축 중 (1) 메모리 성능 저하 문제 발생 → 모니터링 구축 → 원인 규명 → 해결, (2) 서버 다운 시 데이터 유실 → RabbitMQ 도입

**이 경험이 사용된 기업: 4개 전부**

| 구분 | KB증권 | 삼성생명 | 한국자금중개 | 현대해상 |
|------|--------|---------|------------|---------|
| **사용 문항** | 차별화된 역량 | 직무 관련 강점 | 지원동기 | 지원직무 경험 |
| **소제목** | "근본을 파고드는 몰입, 신뢰를 지키는 힘" | "끈기로 구성한 안정적인 운영, 팀원과 함께 이룬 성장" | "IT로 안심과 신뢰를 만들어가겠습니다" | "제가 받았던 안심, 나누며 보람을 느끼고 싶습니다" |
| **사용한 사실** | 메모리 문제 + 모니터링 구축 + HNSW 인덱스 원인 규명 | 메모리 문제 + 모니터링 구축 + Qdrant 설정 조정 | RabbitMQ 도입 + DLQ 적용 (메모리 문제는 축약) | RabbitMQ + 모니터링 둘 다 |
| **강조한 것** | 개인의 집요한 몰입, 정량적 지표 분석, 기술적 깊이 | 팀원과 관점 공유, 협업 속 해결, "혼자였다면 불가능" | "서버 다운 ≠ 문제, 데이터 비영속성 = 문제" 본질 재정의 | 사용자 관점, "기술의 가치는 사용자가 안심할 수 있을 때" |
| **생략한 것** | 팀 협업 과정 | HNSW 등 기술 디테일 | 모니터링/메모리 부분 (간략 언급만) | 개인 몰입 서사, 기술 깊이 축소 |
| **기술 디테일 수준** | 최고 (HNSW 인덱스, 메모리 상주 구조) | 중간 (Qdrant, Prometheus, Grafana) | 중간 (RabbitMQ, DLQ) | 낮음 ("벡터 DB의 인덱스", "메시지 큐") |
| **주체** | "저는" 중심 (90%) | "저는" + "팀원과 함께" (60:40) | "저는" 중심 (80%) | "저는" 중심 + 사용자 관점 (70:30) |
| **문제 정의 방식** | "시스템 신뢰성을 해치는 상황" | "근본적인 해결이 어렵다고 판단" | "문제의 본질이 서버 다운이 아니라 데이터의 비영속성" | "사용자 작업과 데이터가 연속성을 보장하지 못하는 문제" |
| **결과 해석** | "시스템의 신뢰성을 높이는 기반" | "협업 속에서 완성" | "사용자가 안심할 수 있는 환경" | "'안심'의 가치를 기술로 실현" |
| **매칭 가치** | 전문성 + 신뢰정직 | 도전 + 협업 + 성장 | 신뢰 + 안정성 + 본질 파악 | 안심 + 고객 중심 |

#### 핵심 교훈
- **같은 프로젝트에서 2가지 다른 문제(메모리/데이터 유실)를 꺼내 쓸 수 있다**
- 기업이 "전문성"을 원하면 기술 디테일을 높이고, "협업"을 원하면 팀 서사를 늘린다
- 문제 정의 문장 하나만 바꿔도 전체 톤이 달라진다

### 경험 B: 해커톤 갈등 해결 (구름톤 Univ)

> **사실**: 해커톤에서 '업무왕 선정 기준'을 두고 의견 충돌 → 객관적 분석으로 합의 → 수상

**이 경험이 사용된 기업: KB증권, 현대해상**

| 구분 | KB증권 | 현대해상 |
|------|--------|---------|
| **사용 문항** | 힘들었던 과제와 극복 | 서로 다른 의견을 가진 사람들과 목표 달성 |
| **소제목** | "갈등을 기회로, 협업을 성과로" | "다양성을 존중하며 합의를 이끌어낸 도전과 성장" |
| **강조한 것** | 문제 해결 주도, 객관적 분석, 팀 신뢰 구축 | 다양성 존중, 경청, "틀림이 아니라 다양성" |
| **나의 역할 묘사** | "문제 해결에 나섰습니다" (주도적) | "각 팀원의 의견을 끝까지 들으며" (경청형) |
| **갈등 해석** | "기회" → 전화위복 | "다양성의 표현" → 존중 |
| **분석 방법** | "장단점을 객관적으로 분석" | "표로 정리해 공유, 기획 의도 부합도 검토" |
| **결과 해석** | "팀워크로 극복, 협업의 진정한 힘 체감" | "존중과 협력을 바탕으로 가능했던 성과" |
| **마무리 연결** | "협업과 책임감 → 안정적 시스템 구축" | "차이를 존중, 전문성을 인정 → 안정적 IT 서비스" |
| **매칭 가치** | 동반성장 + 신뢰 | 존중 + 공동체 + 협업 |

#### 핵심 교훈
- **같은 갈등인데 "기회"로 볼 것인가 "다양성"으로 볼 것인가의 차이**
- KB증권: "내가 주도해서 해결" / 현대해상: "함께 경청하며 합의"
- 소제목에서 이미 앵글이 결정된다

---

### 경험 C: 대학 동기에게 받은 자극 (성장 서사)

> **사실**: "안주하던 나"를 깨뜨린 동기의 존재 → 태도 전환

**이 경험이 사용된 기업: 삼성생명, 한국자금중개**

| 구분 | 삼성생명 | 한국자금중개 |
|------|---------|------------|
| **사용 문항** | 성장과정 (1500자) | 성장과정 (800자) |
| **분량** | 전체의 절반 이상, 2파트 구성 | 절반 미만, 1파트로 압축 |
| **소제목** | "무모해 보였던 동기, 안주했던 내 모습" + "더 깊이 파고들어, 한계를 깨부수다" | "무모해 보였던 동기, 안주했던 내 모습" |
| **성장 서사 이후 실증** | Nginx upstream keepalive 최적화 (기술적 실증) | "작은 개선이 시스템 전체의 안정성과 가용성을 향상" (추상적 귀결) |
| **강조한 것** | "더 안정적이고 나은 방법" 탐색, 한계 확장 | "끝까지 물고 늘어져 더 나은 성과", 안정성 + 가용성 |
| **마무리 연결** | "어디서든 한계를 뛰어넘고 서비스를 견고하게" | "한국자금중개 IT 업무에서도 한계를 뛰어넘는 가치" |
| **매칭 가치** | 도전 + 성장 + 전문성 | 도전 + 안정성 |

#### 핵심 교훈
- **글자수에 따라 깊이가 달라진다**: 1500자면 기술 실증까지, 800자면 태도 귀결로 마무리
- 같은 소제목이라도 뒤에 오는 실증의 종류가 기업 가치에 맞춰 달라진다

---

### 경험 D: 과 대표 네트워킹 행사

> **사실**: 코로나 후 침체된 학과 → 전례 없는 행사 기획 → 매년 이어지는 전통

**이 경험이 사용된 기업: 현대해상만**

| 구분 | 현대해상 |
|------|---------|
| **사용 문항** | 새로운 방법이나 시도로 긍정적 결과 |
| **소제목** | "누구도 해보지 않은 일, 제가 도전했습니다" |
| **강조한 것** | 공동체 문제를 "내 문제"로 인식, 과감한 도전, 전통으로 정착 |
| **매칭 가치** | 공동체 + 도전 + 책임 |

#### 핵심 교훈
- **비기술 경험도 기업 가치에 정확히 맞으면 S급이 된다**
- 현대해상의 "공동체적 안심 실현"과 완벽히 맞는 경험

### 경험 E: Docker/CI-CD 인프라 개선

> **사실**: Docker Network 도입(보안), 쉘 스크립트 표준화(배포), GitHub Actions CI/CD

**이 경험이 사용된 기업: KB증권만**

| 구분 | KB증권 |
|------|--------|
| **사용 문항** | 핵심가치 (전문성) |
| **소제목** | "전문성으로 안정성을 설계하다" |
| **구성** | 3개 문제 → 3개 해결을 순차적으로 전개 |
| **강조한 것** | 프로세스 개선, 오류 가능성 ↓, "트랜잭션과 트래픽을 안정적으로 처리" |
| **매칭 가치** | 전문성 + 안정성 |

#### 핵심 교훈
- **기술 경험 3개를 묶어서 "전문성"이라는 하나의 스토리로 구성**
- KB증권의 "전문성" 문항이니까 기술 디테일을 최대치로 올림

---

### 경험 F: 지원동기 (개인 서사)

> 지원동기는 기업마다 완전히 다른 서사를 사용

| 기업 | 개인 서사 | 연결 논리 |
|------|----------|----------|
| **KB증권** | 군 복무 중 M-able로 처음 주식 → 개발 공부 후 다시 본 MTS의 기술적 완성도 | "처음 투자를 접하게 해준 플랫폼을 직접 만들고 싶다" |
| **삼성생명** | "주변의 응원과 지지로 안심을 얻으며 성장" | "안심이 성장을 가능하게 한다 → 생명보험의 본질과 맞닿아 있다" |
| **한국자금중개** | 프로젝트에서 안정성과 신뢰성 기반 시스템 구축에 가치를 둬왔음 | "이러한 가치가 회사의 방향성과 맞닿아 있다" |
| **현대해상** | "주변의 응원과 지지로 안심을 얻으며 성장" (삼성생명과 동일 서사) | "안심이 성장을 가능하게 한다 → 손해보험의 본질과 공감" |

#### 핵심 교훈
- **지원동기만큼은 기업의 사업 본질과 직접 연결되는 개인 서사가 필요**
- 삼성생명/현대해상은 같은 "안심" 서사를 썼지만, 연결 대상이 다름 (생명보험 vs 손해보험)
- KB증권은 실제 사용자 경험 → 기술적 감탄 → 만들고 싶다 (가장 구체적)

## 3. 앵글 전환 실전 적용 규칙

### 3.1 기업 가치 유형별 전환 방향

| 기업 가치 유형 | 강조할 것 | 축소할 것 | 기술 디테일 |
|---------------|----------|----------|-----------|
| **전문성/신뢰 중심** (KB증권형) | 개인의 몰입, 정량적 분석, 근본 원인 추적 | 팀 협업 | 최대 (구체적 기술명, 동작 원리) |
| **성장/협업 중심** (삼성생명형) | 팀 협업, 끈기, 안주→도전 전환 | 기술 디테일 | 중간 (도구명은 쓰되 원리는 축약) |
| **본질/구조 중심** (한국자금중개형) | 문제 본질 재정의, 구조적 해결 | 감정 서사 | 중간 (구조적 선택 이유 중심) |
| **공동체/안심 중심** (현대해상형) | 사용자/고객 관점, 공동체 의식 | 개인 영웅담 | 낮음 (사용자 영향 중심) |

### 3.2 같은 경험을 다른 기업에 맞추는 4단계

```
[Step 1] 기업의 암묵적 가치 코드 확인
    ↓
[Step 2] 경험에서 해당 가치와 공명하는 요소 선택
    ↓
[Step 3] 공명하지 않는 요소는 축소 또는 생략
    ↓
[Step 4] 문제 정의/결과 해석 문장을 기업 가치 언어로 교체
```

### 3.3 절대 바꾸면 안 되는 것 vs 바꿔야 하는 것

| 절대 고정 (Fact) | 자유롭게 전환 (Angle) |
|-----------------|---------------------|
| 프로젝트명, 기간, 역할 | 소제목 |
| 실제 사용한 기술명 | 문제 정의 관점 |
| 정량적 결과 수치 | 강조/생략 배분 |
| 실제 행동 내용 | 결과 해석 방식 |
| | 기술 디테일 깊이 |
| | "나" vs "팀" 비중 |
| | 마무리 연결 방향 |

## 4. 이 매핑을 Episode에 적용하는 법

자소서 작성 시 다음 순서로 적용:

```
1. MASTER_INDEX.md에서 Episode 선택
2. 이 문서에서 해당 기업 가치 유형 확인 (3.1)
3. Episode의 사실 중 해당 가치와 공명하는 요소 선택 (3.2)
4. 강조/생략/깊이/주체를 결정
5. 문제 정의 문장과 결과 해석 문장을 기업 가치 언어로 작성
6. generic_storytelling_patterns.md의 구조에 매핑
7. APPENDIX_C의 기업별 오버레이 적용
```
